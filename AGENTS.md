# AGENTS.md

默认用中文回复。除非用户明确要求，不要切换成英文。

这个项目按“原始资料 -> 过程整理 -> 用户确认 -> 正式交付”的方式工作。不要把 Codex 当聊天框用，要把它当一个会读目录、留过程、产出文件的协作工程来用。

## 目录规则

项目里固定使用这些目录：

```text
raw/          原始资料
docs/         过程文档
output/       最终交付物
memory-bank/  长期背景
assets/       可复用素材
```

### raw/

`raw/` 只放原始资料，比如客户文件、PDF、Word、Excel、PPT、截图、会议记录、原始数据、产品资料。

规则：

- 不改写 `raw/` 里的文件。
- 不重命名 `raw/` 里的文件。
- 不移动 `raw/` 里的文件。
- 不用新文件覆盖 `raw/` 里的文件。
- 读到的重要信息，要在 `docs/source-notes.md` 里写清来源文件名。

如果资料里没有，就写“资料未提供”，不要自己补。

### docs/

`docs/` 放工作过程，不放最终成品。

常用文件：

```text
docs/source-notes.md      资料来源、摘录、关键数据
docs/analysis-notes.md    分析、判断、结构、草稿
docs/questions.md         需要用户确认的问题
docs/plan.md              执行计划
```

复杂任务先写 `docs/`，再出成品。复杂任务包括：报告、方案、PPT、表格分析、PDF、图文内容、视频脚本、涉及多个资料文件的任务。

### output/

`output/` 只放最终交付物。

可以放：

- `.docx`
- `.xlsx`
- `.pptx`
- `.pdf`
- 图片
- 视频
- 发布文案
- 压缩包

不要把过程稿、未确认稿、测试文件放进 `output/`。

### memory-bank/

`memory-bank/` 只写以后还会复用的信息。

可以写：

- 项目背景
- 用户长期偏好
- 固定口径
- 已确认结论
- 下次继续时必须知道的进度

不要写：

- 一次性草稿
- 临时分析
- 未确认猜测
- 密码
- Token
- API Key
- 客户隐私和敏感数据

### assets/

`assets/` 放长期复用素材，比如 Logo、品牌图、模板图、稳定图标。

临时截图、候选图、实验图先放 `output/`。只有确定会长期复用，才放进 `assets/`。

## 工作顺序

简单任务可以直接做。简单任务指：改一句话、整理一小段文字、回答一个明确问题、改一个确定文件里的小问题。

复杂任务按下面顺序做：

1. 先看 `raw/`。
2. 再看 `docs/`。
3. 再看 `memory-bank/`。
4. 把资料来源写到 `docs/source-notes.md`。
5. 把结构、判断、草稿写到 `docs/analysis-notes.md`。
6. 把缺失信息写到 `docs/questions.md`。
7. 等用户确认关键方向。
8. 再生成 `output/` 里的正式文件。

用户明确说“直接生成”时，可以简化过程，但不能跳过事实检查、目录规则和文件验证。

## 什么时候先问用户

遇到这些情况，先问，不要猜：

- 不知道最终要交付什么文件。
- 原始资料不够。
- 资料之间互相矛盾。
- 关键数字缺失。
- 客户名称、时间、范围、版本不明确。
- 需要用户在多个方案中选一个。
- 正式 Word、Excel、PPT、PDF 的大纲还没确认。
- 需要联网查资料，但用户没有说明是否允许使用外部资料。

提问要短。一次最多问 3 个真正影响结果的问题。

## 资料和事实规则

信息优先级：

1. 用户当前消息。
2. `raw/` 原始资料。
3. `docs/` 已整理过程。
4. `memory-bank/` 已确认背景。
5. 用户允许后查到的外部资料。

禁止编造：

- 客户名称
- 金额
- 增长率
- 市场规模
- 政策条款
- 引用来源
- 调研结论
- 用户没有确认过的判断

如果使用外部资料，必须记录：

```text
来源名称：
链接或文件名：
引用日期：
```

不确定的信息统一标记：

```text
待确认：
```

不要把“待确认”的内容写成确定结论。

## Skill 使用规则

涉及正式文件时，用对应 Skill，不要只在聊天里写内容。

```text
minimax-docx   Word / DOCX
minimax-xlsx   Excel / XLSX / CSV / 表格分析
minimax-pdf    PDF / 正式报告
ppt-master     PPT / PPTX / 演示文稿
```

规则：

- 用户要 Word，就生成 `.docx`。
- 用户要 Excel，就生成 `.xlsx`。
- 用户要 PPT，就生成 `.pptx`。
- 用户要 PDF，就生成 `.pdf`。
- 当前环境没有对应 Skill 时，直接说明缺少什么，再给可执行的替代方案。

`superpowers` 用来管流程，不负责替代文件生成。适合用在：

- 需求还不清楚时拆问题。
- 任务比较大时写计划。
- 执行中需要分阶段检查。
- 完成前做自检。

## 文件命名

输出文件用这个格式：

```text
YYYYMMDD-主题-v1.ext
```

示例：

```text
20260706-sales-analysis-v1.xlsx
20260706-project-report-v1.docx
20260706-client-proposal-v1.pptx
20260706-final-report-v1.pdf
```

不要使用：

```text
新建文档.docx
最终最终版.pptx
未命名.xlsx
111.pdf
```

多次修改时用 `v1`、`v2`、`v3`。不要覆盖旧版本，除非用户明确要求。

## 交付前检查

回复完成前，检查这些事：

- 文件是否真的生成。
- 文件是否在 `output/`。
- 是否误改了 `raw/`。
- 关键数字是否有来源。
- 是否还有 `待确认` 内容。
- 是否使用了正确的 Skill。
- 是否有临时文件需要删除。

## 交付时说清楚

完成后回复里必须说明：

1. 生成或修改了哪些文件。
2. 文件路径是什么。
3. 使用了哪些原始资料。
4. 哪些内容需要用户复核。
5. 如果有未完成项，直接说。

不要只说“已完成”。

## 禁止事项

- 不要覆盖 `raw/` 原始资料。
- 不要把正式文件放在项目根目录。
- 不要把未确认稿放进 `output/`。
- 不要把一次性分析写进 `memory-bank/`。
- 不要编造数据、来源、结论。
- 不要把不确定内容写成事实。
- 不要用聊天回复代替用户要的正式文件。
- 不要删除用户文件，除非用户明确要求。
- 不要保存密码、Token、API Key 或敏感凭据。
